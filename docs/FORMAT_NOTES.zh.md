# 三国霸业地图与关卡文件二进制结构笔记

本文记录当前已经从游戏目录、`Emperor.exe`、`kingdom.cel`/`kingdom.atr`、以及 `stage*.m/.s/.x/.stg/.spr/.dor/.evt` 中确认或推断出的格式。结论按“已确认 / 高置信推断 / 待验证”区分，避免把还没证实的逻辑写成事实。

## 总结

`.m` 的每个 cell 不是只有 `acwx/acwy/acwz`。每个 cell 固定 16 字节，目前已拆出 12 个字段：前三个 int16 是三层图块/物件索引，后面的字节包含渲染缓存、对象/地形辅助信息、以及疑似通行或占位相关标志。

但是，`.m` 也不是完整的关卡语义数据库。城池 id、势力、单位、剧情触发、城门/入口等信息，更可能分布在同名不同后缀文件里：`.stg`、`.spr`、`.dor`、`.evt`。`.s`/`.x` 是固定 160x160 的栅格数据，可能是缩略图、寻路、通行、视野或运行时缓存之一，不能简单当作最终大地图。

## `.m` 文件结构

### 文件头

`.m` 文件头目前确认如下：

| 偏移 | 类型 | 含义 | 状态 |
| --- | --- | --- | --- |
| `0x00` | uint32 little-endian | 地图宽度，单位是 cell | 已确认 |
| `0x04` | uint32 little-endian | 地图高度，单位是 cell | 已确认 |
| `0x08` | char[8] | 魔数/版本字符串 `Hello1.0` | 已确认 |
| `0x10` | record[width * height] | cell 记录数组，每条 16 字节 | 已确认 |

文件大小公式：

`16 + width * height * 16`

例如 `stage11.m`：宽 42，高 168，总 cell 数 7056，文件大小为 `16 + 42 * 168 * 16 = 112912` 字节。

### cell 记录结构

每个 cell 固定 16 字节，小端序。当前字段如下：

| cell 内偏移 | 类型 | 暂定字段名 | 当前理解 | 状态 |
| --- | --- | --- | --- | --- |
| `+0x00` | int16 | `acwx` | 地表基础层图块索引；通常每格都有非负值 | 已确认 |
| `+0x02` | int16 | `acwy` | 地表叠加/边缘/过渡层索引；`-1` 表示无 | 已确认 |
| `+0x04` | int16 | `acwz` | 物件/建筑/树木/城池部件索引；`-1` 表示无 | 已确认 |
| `+0x06` | int16 | `word06` | 目前样本中多为 0 | 已确认存在，语义待定 |
| `+0x08` | uint8 | `byte08` | 稀有非零辅助标志，可能与特殊地形/对象相关 | 待验证 |
| `+0x09` | uint8 | `byte09` | 常见 0/1 标志；是通行/地形/占位候选字段 | 高优先级待验证 |
| `+0x0a` | uint8 | `byte10` | 对象/地形类别候选；EXE 中存在对 `0x65..0x6e` 的比较 | 高置信推断 |
| `+0x0b` | uint8 | `byte11` | 子图块/footprint 子索引候选；部分关卡里常以 36 格为一组出现 | 高置信推断 |
| `+0x0c` | uint8 | `byte12` | 大多为 0 | 语义待定 |
| `+0x0d` | uint8 | `final_palette` | 最终渲染/小图/缓存相关调色板字节候选 | 高置信推断 |
| `+0x0e` | uint8 | `byte14` | 目前样本中多为 0 | 语义待定 |
| `+0x0f` | uint8 | `byte15` | 目前样本中多为 0 | 语义待定 |

因此编辑器不应该只保存三层数字。即使当前 UI 先开放 `acwx/acwy/acwz`，内部数据模型也必须保留完整 16 字节记录，否则保存后会丢失通行、对象 footprint、缓存或其他逻辑字段。

## `acwx/acwy/acwz` 如何构成画面

### `acwx`：基础地形层

`acwx` 是每个 cell 的基础地形图块，几乎所有 cell 都有非负索引。它从 `kingdom.cel` 的 `acwx` 资源组取图，绘制为 40x20 的等距菱形地块。

当前从 EXE 和图像对齐结果确认的世界坐标到屏幕坐标为：

`screen_x = col * 40 + (row 为奇数 ? 20 : 0)`

`screen_y = row * 10`

这意味着地图不是一个传统完整菱形画布，而是按奇偶行错位的 staggered isometric 网格铺开。

### `acwy`：地表叠加/过渡层

`acwy` 同样使用 40x20 菱形图块。它通常不是每格都有，`-1` 表示该格没有叠加层。视觉上它更像道路、水岸、边缘、地形过渡或附加地表细节。

### `acwz`：对象/建筑层

`acwz` 是可高出地表的对象层，包含树、建筑、城池部件等。`acwz` 图块不是固定 40x20，而是带有独立头部：

| 偏移 | 类型 | 含义 | 状态 |
| --- | --- | --- | --- |
| `+0x00` | uint32 | x anchor | 已确认用于绘制偏移 |
| `+0x04` | uint32 | y anchor | 已确认用于绘制偏移 |
| `+0x08` | uint32 | width | 已确认 |
| `+0x0c` | uint32 | height | 已确认 |
| `+0x10` | uint32 | pointer/unknown | 存在，语义待定 |
| `+0x14` | bytes | 像素数据 | 已确认 |

EXE 中的对象绘制逻辑等价于：

`dest_x = screen_x - x_anchor`

`dest_y = screen_y - y_anchor`

当前复原渲染器为了和实际游戏截图对齐，使用地块中心作为对象锚点：

`dest_x = tile_screen_x + 20 - x_anchor`

`dest_y = tile_screen_y + 20 - y_anchor`

这解释了为什么单纯把 `acwz` 垂直居中会“看起来接近”，但并不是完整规则。真正规则必须同时考虑 anchor、footprint、以及绘制顺序。

## 旁路文件与“城池 id”等语义数据

`.m` 的 `acwz` 可以画出城池图形，但图形不等于城池实体。城池 id、所属势力、可进入状态、驻军、事件触发等语义信息，理论上不应该只靠 `acwz` 保存；从文件结构看，它们更可能在以下同名文件中。

### `.stg`：关卡/剧本/城池语义候选

观察到 `.stg` 是变长文件。多个样本开头有稳定 dword：第一个常见为 `1`，第二个常见为 `76`，个别如 `stage00` 为 `72`。文件中还能看到文本内容。

当前推断：`.stg` 很可能保存关卡级配置，例如剧本数据、城池列表、势力/阵营、AI 或初始状态。是否直接保存“城池 id -> 地图坐标/归属”的表，还需要继续对照城池位置和 EXE 访问逻辑确认。

进一步确认：按 `cp950/big5` 解码，`0x08` 附近可以直接读出剧本标题。例如 `stage00.stg` 是“`三國霸業`”，`stage11.stg` 是“`練習一劉備出發`”，`stage20.stg` 是“`為圖長久取四郡`”。另外 `0x24/0x28` 两个 dword 在大量样本里分别呈现 `184/187`、`206/212` 这类年份范围，极像剧本起止年份候选字段。

继续用 `decoded_strings_preview` 和 76 字节 record 预览做比对后，可以直接扫到 `平原`、`荊州`、`襄陽`、`南郡`、`劉備`、`關羽`、`諸葛亮`、`步兵`、`弓箭兵`、`中立國家` 等字符串。它已经不像单纯的标题文件，更像“剧本实体表 + 名称表”的混合体：至少承载了城池、武将、兵种、势力/归属这一层语义。按当前 76 字节预览看，不同记录偏移承载的字符串类型不同，说明同一张表里很可能混合了多种子记录类型。

### `.spr`：单位/士兵数据

`.spr` 文件头为 ASCII：

`Soldier Data`

其后出现稳定的 dword，例如 `180`、`36`，并跟随变长记录。当前高置信推断它保存单位、士兵、武将或地图上动态角色的初始数据。

进一步确认：`stage01.spr` 在这 20 字节左右的 meta 前缀之后几乎全部为 0，说明 `.spr` 可以为空，单位层并不是每张图都必须有实例数据；而 `stage11.spr`、`stage20.spr` 同一位置之后存在大量非零 payload，说明它更像“可选单位/驻军初始表”。当前还不能把 36 字节实体记录完全锁死，但“单位数据不在 `.m` 里，而是在 `.spr` 里按关卡单独提供”这一点已经比之前更明确。

这类数据不应该写入 `.m` cell；地图编辑器后续需要单独做“单位/对象实体层”编辑。

### `.dor`：门/入口/通道数据

`.dor` 文件头为 ASCII：

`Door    Data`

其后常见 dword `56` 或 `60`。从命名和结构推断，它可能保存城门、入口、通道、关隘或传送/进城点数据。

进一步确认：`stage11.dor` 只有 28 字节，并且 16 字节头后的 body 全 0，说明 `.dor` 也可能为空或只保留文件级头；`stage20.dor`、`stage01.dor` 则都有明显非零 payload。也就是说，“门/入口层”看起来和单位层一样，是按关卡可选启用的语义 sidecar，而不是每张地图固定存在的强制表。

如果城池有“可进入点”“门是否开启”“入口坐标”等信息，`.dor` 是优先调查对象。

### `.evt`：事件/脚本数据

`.evt` 是事件/脚本文件。第二个 dword 在样本中稳定为 `72`，并且 `stage01.evt` 到 `stage04.evt` 可以被精确解释成 `8` 字节头加 `251` 条 `72` 字节记录；其他关卡则常见“主记录表 + 小尾巴”结构。当前已提取到的 ASCII 命令词包括 `talk`、`VIEW`、`MAP`、`MAPALL`、`MOVE`、`OPEN`、`TIME`、`TIMEOVER`，进一步支持它是脚本命令层。

继续扫描到的 `cp950` 字符串还包括 `勝利`、`失敗`、`提示`、`佔領四郡`、`敵兵全死`、`荊州被佔領`、`黃忠加入提示`、`劉備出現`、`到達涿縣` 等。也就是说，`.evt` 不只是 opcode 表，还夹带目标名、提示文案、对话标签与条件名。按 72 字节 record 预览看，这些文本会在记录内部不同偏移滚动出现，更像“命令记录 + 文本参数/文本块引用”的混合结构。

地图编辑器不应把事件触发直接等同于 `.m` 的图块。后续应把 `.evt` 做成独立事件层，并通过坐标或对象 id 与地图关联。

## `.s` / `.x` 与小地图、通行性的关系

`.s` 和 `.x` 文件都是 25600 字节，也就是 `160 * 160` 的 1 字节栅格。它们不像 `.m` 一样保存变尺寸 cell 表，而是固定尺寸逻辑图。

当前已知事实：

- `.s`：多数关卡存在，大小固定 25600 字节。
- `.x`：多数关卡存在，大小固定 25600 字节。
- 它们更像游戏内部用的固定分辨率地图缓存、缩略图、通行/寻路表、战争迷雾或地形分类表。
- 不能确认二者谁是小地图、谁是通行表；需要从 `Emperor.exe` 对 `.s`/`.x` 的读写逻辑继续验证。

编辑器设计上要预留一个“派生缓存重建”步骤：修改 `.m` 后，不能只更新大地图画面，也要能重建或同步 `.s`/`.x`，否则游戏里可能出现缩略图、寻路或通行信息与画面不一致。


### 当前相关性结论（2026-06-24）

使用 `tools/analyze_stage_sidecars.py` 对 `stage01`、`stage11`、`stage20`、`stage29` 做统计后，当前可以再往前确认几件事：

- `.s` 和 `.x` 都是固定 25600 字节，但二者并不是完全不同的两份数据；四个样本的逐字节相同率分别为 86.31%、80.94%、79.57%、72.71%。
- `.x` 与 `.m` 的 `final_palette` 唯一值集合重叠度 consistently 高于 `.s`。例如 `stage11` 中，`.s` 与 `final_palette` 重叠 62 个唯一值，而 `.x` 重叠 76 个。
- 这说明 `.s` / `.x` 更像同一逻辑平面的两个派生栅格、缓存或不同阶段结果，而不像彼此无关的原始资源文件；其中 `.x` 目前看起来更接近最终缩略图或颜色结果。
- 采样关卡中，`.s` 从未出现“某像素非 `240`，但 `.x` 却是 `240`”的情况；反过来 `.x` 还会比 `.s` 多出 `24` 到 `190` 个非 `240` 像素。结合二者大部分非 `240` 像素值仍然相同，这更像“`.x` 是较完整的颜色缓存，`.s` 是更保守的遮罩/裁剪伴随层”。
- `Emperor.exe` 的字符串上下文中，`.s`、`.x`、`.m` 紧邻出现；而 `.evt`、`.dor`、`.spr` 则与 `stage01.evt`、`stage01.dor`、`castle.magic.txt`、`soldier.txt`、`general.txt` 相邻，进一步支持“地图主表”和“旁路语义文件”分工。

## 通行性数据的候选位置

目前不能断言“是否可通行”只在一个字段中。候选位置按优先级如下：

| 候选 | 证据 | 风险 |
| --- | --- | --- |
| `.m` 的 `byte09` | 常见 0/1 分布，像布尔标志；`stage11` 中 1 的数量为 1688/7056 | 也可能是渲染或地形类别标志 |
| `.m` 的 `byte08/byte10/byte11` | 有特殊值和成组分布，`byte10` 被 EXE 与 `0x65..0x6e` 比较 | 可能表示对象 footprint、地形类别或城池占位，不一定直接等于通行 |
| `.s`/`.x` | 固定 160x160 栅格非常适合存通行/寻路/小地图缓存 | 目前未确认 EXE 使用语义 |
| `.dor` | 门/入口类数据通常会影响通行 | 可能只影响特殊入口，不负责普通地形 |

下一步验证方式：

1. 在 EXE 中定位 `.s`/`.x` 文件名引用和加载函数。
2. 找到寻路、鼠标点击、单位移动或建筑占位相关代码，看它读取哪一份数据。
3. 用已知河流、山地、城墙、道路位置对照 `byte09`、`byte10`、`.s`、`.x` 的空间分布。
4. 在复制关卡上做最小改动，进入游戏验证单位是否能通过。

## 编辑器设计影响

1. `.m` 编辑器内部必须保留完整 16 字节 cell 记录。UI 可以先只显示 `acwx/acwy/acwz`，但保存时要原样带回未编辑字段。
2. 资源面板需要同时展示 `acwx/acwy/acwz` 的图形预览。选择资源时，写入的是索引；画面重绘时，从 `kingdom.cel` 对应资源组取图。
3. 城池、单位、门、事件不应作为 `acwz` 的附属字符串硬塞进 `.m`。后续应分别解析 `.stg/.spr/.dor/.evt`，做成独立可见图层。
4. 通行性编辑要等字段确认后再开放写入。短期可以做“只读 overlay”：把疑似通行字段叠加显示，帮助人工验证。
5. 修改 `.m` 后，还需要设计 `.s`/`.x` 的同步策略。最保守方案是先导出补丁和新 `.m`，暂不改 `.s`/`.x`；等 EXE 逻辑确认后再自动重建。

### 记录家族层（2026-06-24）

继续对 `.stg/.evt` 做 record-family 聚类后，现在可以再往前走一步：它们不是“同一种固定记录反复出现”，而是“同一条固定步长里混合了多种子模板”。

- `.stg` 中，当前最强的三类提示字是 `224`、`92`、`96`。
- 带 `224` 的家族，基本都落在武将、兵种、盗贼这类实体记录上；名字字段会在 `12/24/32/44/64` 等多个偏移滚动出现。
- 带 `92` 的家族，基本都落在城池、城市、军寨、山寨这类地点/据点记录上。
- 带 `96` 的家族，目前更像势力、君主、归属类记录，例如 `劉備 + 禤a`、`中立國家`。
- 这说明 `.stg` 不是单纯“按记录号查表”，而更像多个 76 字节子模板共用一个容器。后续应按家族分别拆 id、势力、坐标。

- `.evt` 里也出现了明显家族。`stage01.evt` 中，大量姓名记录会按 `0/8/16/24/32/48/56/64` 等偏移滚动出现，同时伴随 `72`、`55`、`189..257` 这类控制字，像是“名字槽位表”或“目标名单表”。
- `stage20.evt` 则更明显分出 `流程`、`提示`、`事件`、`条件/目标` 家族，例如 `佔領四郡`、`荊州被佔領`、`黃忠加入提示`。
- 当前脚本给出的 `candidate_small_u16_fields` 只能算“候选列”，还不能直接当成最终坐标；但它已经能帮助筛出哪些 `u16` 列最像小范围 id、owner 或 stage 内坐标。

### Sidecar 工作簿导出与 `.stg/.evt` 二进制构成（2026-06-24）

为了继续拆 `.stg/.evt` 的字段，本仓库新增了一套“先摊平、再做工作簿”的导出链：

- `tools/export_stage_sidecar_workbook.mjs`：把这些表生成 `derived/sidecar_analysis/stg_evt_analysis.xlsx`。

当前工作簿的关键 sheet：

- `文本槽位`：统计不同家族的文本偏移命中次数，用来观察“文本槽位”是否复用。
- `候选字段`：把 `candidate_small_u16_fields` 展平成逐行数据，便于筛选 `word_index/byte_offset`。
- `stg记录` / `evt记录`：把每条含文本 record 的 `texts_joined`、`text_layout`、`nonzero_words`、`w00..w37/35` 全部摊平。

#### `.stg` 当前可确认的二进制构成

以当前 33 个关卡样本看，`.stg` 可以先按“8 字节文件头 + 76 字节主记录表 + 少量尾巴数据”理解：

| 偏移 | 类型 | 当前理解 |
| --- | --- | --- |
| `0x00` | `u32` | 常见为 `1`，更像版本号或类型号 |
| `0x04` | `u32` | 常见为 `76`，与主记录步长一致 |
| `0x08..` | bytes | 主体记录区；大多数样本按 76 字节步长可拆 |
| 文件尾 | bytes | 常见剩余 `20/44/48/56` 等尾巴字节，说明记录表后还有补充段或索引 |

典型样本：

- `stage01.stg`：`dwords = [1, 76, ...]`，主记录 `2502` 条，尾巴 `48` 字节。
- `stage11.stg`：`dwords = [1, 76, ...]`，主记录 `21` 条，尾巴 `20` 字节。
- `stage20.stg`：`dwords = [1, 76, ...]`，主记录 `584` 条，尾巴 `56` 字节。

当前更像是“文件头之后的第一条 76 字节记录同时承载关卡标题”，而不是“文件头后先有一段独立标题区再开始 record 表”。原因是：

- `title_cp950` 直接从 `0x08` 开始就能读到，例如 `亂世來臨`、`練習一劉備出發`、`為圖長久取四郡`。
- `0x24/0x28` 两个 `u32` 在大量样本中表现得像关卡年份范围，例如 `189..500`、`184..187`、`206..212`。

按当前聚类结果，`.stg` 不是单一 record 模板，而是多个 76 字节子模板混装：

| 家族 | 当前语义 | 跨关卡总量 |
| --- | --- | --- |
| `troop_entry` | 兵种/部队类记录 | `863` |
| `bandit_entry` | 盗贼/山贼类记录 | `245` |
| `faction_or_ruler` | 势力/君主/归属类记录 | `60` |
| `general_entry` | 武将类记录 | `58` |
| `city_or_structure` | 城市/军寨/山寨/据点类记录 | `39` |
| `scenario_title` | 关卡标题类记录 | `19` |

当前最关键的结构特征：

1. 文本槽位明显复用。武将名/标签常见于 `12/17`、`28/33`、`48/53`、`68/73`；势力名/标签常见于 `64/69`、`0/5`、`48/53`。
2. 控制字 `224`、`92`、`96` 很强：`224` 主要落在武将、兵种、盗贼等实体记录；`92` 主要落在城市、军寨、山寨、据点；`96` 更像势力/君主/归属记录。
3. `troop_entry` 的高频候选列集中在 `w30@60`、`w24@48`、`w26@52`、`w32@64`、`w28@56`、`w22@44`；这些列比其它列更像 owner id、兵种 id、局部坐标或同类索引。

因此目前对 `.stg` 的稳妥理解是：头部提供模板信息，主体是 76 字节混合记录容器，文件尾另有少量尚未命名的补充数据。

#### `.evt` 当前可确认的二进制构成

以当前样本看，`.evt` 可以先按“8 字节文件头 + 72 字节命令记录表 + 可选尾巴”理解：

| 偏移 | 类型 | 当前理解 |
| --- | --- | --- |
| `0x00` | `u32` | 小范围计数、脚本段编号或阶段号；不同关卡会变 |
| `0x04` | `u32` | 基本稳定为 `72`，与主记录步长一致 |
| `0x08..` | bytes | 主命令记录区；主要按 72 字节步长解析 |
| 文件尾 | bytes | 常见 `24/32/52/60` 等尾巴字节，像补充参数或小索引区 |

典型样本：

- `stage01.evt`：`dwords = [219, 72, 2, ...]`，可精确拆成 `251` 条 72 字节记录，无尾巴。
- `stage11.evt`：`dwords = [8, 72, 7, ...]`，主记录 `27` 条，尾巴 `52` 字节。
- `stage20.evt`：`dwords = [20, 72, 5, ...]`，主记录 `68` 条，尾巴 `32` 字节。
- `stage45.evt`：`dwords = [25, 72, 2, ...]`，主记录 `127` 条，尾巴 `60` 字节。

`.evt` 里已确认能直接嵌入 ASCII 命令词：`talk`、`VIEW`、`MAP`、`MAPALL`、`MOVE`、`OPEN`、`TIME`、`TIMEOVER`，以及个别自定义 token（如 `MAP1`、`BEN`、`ALEX`、`DANNY`）。这说明它不是纯文本表，而是脚本命令层。

同时，`.evt` 里还嵌有大量 `cp950` 文本，例如：

- 胜负/提示：`勝利`、`失敗`、`提示`
- 目标/条件：`佔領四郡`、`敵兵全死`、`荊州被佔領`
- 事件/提示：`黃忠加入提示`、`劉備出現`、`到達涿縣`

按当前聚类结果，`.evt` 至少混有以下几类 72 字节子模板：

| 家族 | 当前语义 | 跨关卡总量 |
| --- | --- | --- |
| `flow_text` | 流程/步骤文本 | `2128` |
| `text_mixed_record` | 混合文本命令 | `358` |
| `name_slot_72_family` | 名字槽位表（伴随 72/55 控制字） | `182` |
| `name_slot_highid_family` | 高 id 名字槽位表（伴随 55/189+ 控制字） | `102` |
| `condition_or_objective` | 条件/目标文本 | `64` |
| `event_text` | 事件文本 | `31` |
| `prompt_text` | 提示文本 | `9` |

当前最关键的结构特征：

1. `flow_text` 文本偏移极分散，`0/4/8/12/36/40/52/60/64/68` 都高频命中，更像“共享命令骨架里的文本参数槽位”。
2. `name_slot_72_family` 明显带 `72`、`55` 这类控制字，名字会在 `8/16/24/32/40/48/56/64` 之间滑动。
3. `name_slot_highid_family` 则经常伴随 `55`、`189..257` 以及递增 id，看起来像“另一套名字索引表/目标名单表”。
4. `condition_or_objective`、`event_text`、`prompt_text` 也都不是独立字符串块，而是混在 72 字节命令记录中。

因此目前对 `.evt` 的稳妥理解是：头部说明脚本主记录步长为 72，主体是固定步长命令记录表，同一个 72 字节容器内混有多种命令/文本参数模板，尾部存在一小段尚未命名的附加数据。

#### `.stg` 对齐后字段结论（2026-06-24）

进一步把 `.stg` 记录按控制字旋转到同一参考列后，可以把“同模板平移”从现象变成更具体的字段假设。当前使用的对齐规则是：

- `general_entry`：把 `224` 对齐到 `w04`
- `troop_entry`：把 `224` 对齐到 `w00`
- `faction_or_ruler`：把 `96` 对齐到 `w00`
- `city_or_structure`：把 `92` 对齐到 `w02`

这一步的意义在于：很多原本散落在 `0/8/12/24/28/48/64/68` 等槽位的名字、标签和控制字，旋转后会落回同一套列，说明它们不是不同模板，而更像同一个模板沿着 record 做循环平移。

##### `general_entry`

对齐后，下面这组列已经相当稳定：

- `w04 = 224`：武将实体家族锚点
- `w06/w07`：武将名字的两个 `u16` 字块
- `w08/w09 = 23040 / 20145`：几乎恒定，对应 `Z將` 这类标签块
- `w02`：强烈疑似“所属势力 id”

当前最强证据是把 `.stg faction_or_ruler` 对齐后的势力编号拿来对照：

- `劉備` 阵营武将行里，`w02` 大量为 `1`
- `曹操` 阵营武将行里，`w02` 大量为 `2`
- `孫堅` 阵营武将行里，`w02` 大量为 `3`
- `劉焉` 阵营武将行里，`w02` 大量为 `4`
- `馬騰` 阵营武将行里，`w02` 大量为 `6`
- `袁紹` 阵营武将行里，`w02` 大量为 `7`

因此，当前可以把 `general_entry.w02` 记为“高置信度的势力编号候选列”。

`w16/w18` 也经常出现小值，但目前还不能稳定命名。它们更像人物 id、驻扎点 id、脚本引用 id 或其它武将属性编号，而不是简单 owner。

##### `faction_or_ruler`

对齐后，下面这组列最稳定：

- `w00 = 96`：势力/君主家族锚点
- `w02/w03`：势力名或君主名的两个 `u16` 字块
- `w04/w05 = 59904 / 25006` 或 `60080 / 25006`：常见标签块，对应 `禤a` / `國家` 一类尾部文本
- `w12`：高置信度的势力编号候选列

当前样本中已经能直接读出：

- `曹操 | 禤a` -> `w12 = 2`
- `孫堅 | 禤a` -> `w12 = 3`
- `劉焉 | 禤a` -> `w12 = 4`
- `馬騰 | 禤a` -> `w12 = 6`
- `袁紹 | 禤a` -> `w12 = 7`
- `董卓 | 禤a` -> `w12 = 8`

这也正好解释了上一节里 `general_entry.w02` 的对照关系。

##### `troop_entry`

对齐后，`troop_entry` 比前两类更干净，已经能看出明显的兵种编码对：

| 文本 | `w12` 候选码 | `w14` 候选码 |
| --- | --- | --- |
| `步兵` | `219` | `116` |
| `槍兵` | `220` | `117` |
| `騎兵` | `221` | `118` |
| `弓箭兵` | `222` | `119` |
| `水兵` | `223` | `120` |
| `投石車` | `224` | `121` |

这组值是目前最漂亮的一条“字段级”证据：`w12/w14` 很可能分别对应兵种内部编号与另一套资源/显示编号。

另外，`w24` 常见 `0/1`，`w26` 常见 `0/50`，`w36` 常见 `1/2/3`，它们也像模板内的小枚举位，但当前还缺少足够语义证据，不宜提前命名。

##### `city_or_structure`

对齐后，`city_or_structure` 里至少能分出“完整城市行”和“军寨/山寨行”两类。

对于带 `W城市` / `W城市3` 标签的完整城市行，当前最像“地点编号”的列是 `w14`。已知样本：

- `鄴 | W城市` -> `w14 = 7`
- `譙 | W城市` -> `w14 = 18`
- `宛 | W城市` -> `w14 = 20`

这些值很像城市/据点 id，而不是 owner 或坐标。

`w18` 虽然也是小值，但目前拿它和 `faction_or_ruler.w12` 对照时并不稳定。例如：

- `鄴` 的 `w18 = 4`
- `譙` 的 `w18 = 2`
- `宛` 的 `w18 = 3`

这组三个值目前并不能可靠解释成城市 owner，所以 `w18` 暂时只能记为“小范围枚举字段候选”，不能直接命名为所属势力。

#### 当前这一轮对齐分析的意义

这轮最重要的收获不是“所有列都命名完了”，而是确认了三件关键事实：

1. `.stg` 的很多家族确实存在“同一模板循环平移”的现象，不应再按“完全不同记录结构”去拆。
2. `general_entry.w02` 和 `faction_or_ruler.w12` 已经可以互相对证，形成第一条高置信度跨家族字段关系。
3. `troop_entry.w12/w14` 已经形成稳定兵种编码对，可直接作为后续编辑器或数据字典的基础候选。

#### 当前最值得继续追的字段方向

#### 当前最值得继续追的字段方向

基于工作簿和当前二进制拆解，后续优先级建议如下：

1. `.stg city_or_structure`：优先找城市 id、owner id、地图坐标。
2. `.stg faction_or_ruler`：优先找势力编号、君主编号、所属关系。
3. `.stg general_entry`：优先找武将 id、所属势力、出生或驻扎位置。
4. `.evt name_slot_72_family / name_slot_highid_family`：优先确认它们到底是“角色名单”“目标名单”还是“说话人槽位”。
5. `.evt condition_or_objective`：优先确认坐标型字段、目标对象 id、触发条件码。

当前结论仍然是“结构级确认”，不是最终字段命名；但已经足够支持下一轮按家族拆列，而不必再从整个文件盲扫。


#### `stage.ini`：全局母表（2026-06-24）

继续拆解后可以确认，\stage.ini\ 并不是文本 ini，而是一个“主表 + 尾表”的二进制母库。

##### 主表

- 文件头前 8 字节可直接读成：\main_count=277\、\main_stride=224\。
- 也就是说，文件头之后首先是 \277\ 条、每条 \224\ 字节的固定步长记录。
- 当前高置信理解：这部分主要是全局武将/角色母表。记录里能直接读出 \劉備\、\關羽\、\張飛\、\趙雲\、\曹操\、\孫權\、\周瑜\、\呂布\ 等名字，且大量记录带有 \Z將\ 标签。

##### 尾表

- 主表之后还有一段 \174 * 76\ 字节的固定步长尾表。
- 这段结构风格与关卡 \.stg\ 很接近，可以继续按 76 字节 record 去做 family 拆分。
- 当前聚类结果：

| 家族 | 数量 | 当前理解 |
| --- | --- | --- |
| \text_mixed_record\ | \120\ | 技能/说明/混合文本字典 |
| \troop_entry\ | \26\ | 全局兵种/部队字典 |
| \city_92_family\ | \18\ | 92 锚点城市记录 |
| \city_or_structure\ | \6\ | 带 \W城市\ 标签的完整城市记录 |
| \bandit_entry\ | \3\ | 盗贼/山寨类字典 |
| \scenario_title\ | \1\ | 场景/标题型混合记录 |

##### 城市 id 的新印证

\stage.ini\ 尾表中的城市名与单关 \.stg city_or_structure\ 已经能多点对证，说明它们共享同一套城市/地点 id 空间。当前高置信样本包括：

- \平原 -> 6\
- \鄴 -> 7\
- \譙 -> 18\
- \宛 -> 20\
- \吳 -> 31\
- \成都 -> 38\

这进一步说明：城市 id 不只存在于某一个 \stageNN.stg\，而是先在 \stage.ini\ 这类全局母表里占位，再被各关卡数据引用。

##### 与 \.evt\ / \.stg\ 的关系

- \.evt name_slot_72_family / name_slot_highid_family\ 中已经能看到大量与 \.stg general_entry.w16\ 对上的武将 id。
- \stage.ini\ 主表又提供了更完整的武将母表，所以当前最稳妥的理解是：
  1. \stage.ini\ 负责全局字典；
  2. \.stg\ 负责关卡内实体/势力/城市/兵种引用；
  3. \.evt\ 负责事件脚本层对这些全局 id 的调用。

- \tools/export_stage_ini_tables.py\：导出 \derived/stage_ini_analysis/stage_ini_tables.json\。
- \tools/export_stage_ini_workbook.mjs\：生成 \derived/stage_ini_analysis/stage_ini_analysis.xlsx\。
