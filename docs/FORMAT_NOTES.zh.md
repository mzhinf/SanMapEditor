# 三国霸业地图与关卡文件二进制结构笔记

本文记录当前已经从游戏目录、`Emperor.exe`、`kingdom.cel`/`kingdom.atr`、以及 `stage*.m/.s/.x/.stg/.spr/.dor/.evt` 中确认或推断出的格式。结论按“已确认 / 高置信推断 / 待验证”区分，避免把还没证实的逻辑写成事实。

## 总结

`.m` 的每个 cell 不是只有 `acwx/acwy/acwz`。每个 cell 固定 16 字节，目前已拆出 12 个字段：前三个 int16 是三层图块/物件索引，后面的字节包含渲染缓存、对象/地形辅助信息、以及疑似通行或占位相关标志。

但是，`.m` 也不是完整的关卡语义数据库。城池 id、势力、单位、剧情触发、城门/入口等信息，更可能分布在同名不同后缀文件里：`.stg`、`.spr`、`.dor`、`.evt`。`.s`/`.x` 是固定 160x160 的栅格数据，可能是缩略图、寻路、通行、视野或运行时缓存之一，不能简单当作最终大地图。

## `.m` 文件结构

### 文件头

`.m` 文件头目前确认如下：

| 偏移 | 类型 | 含义 | 状态 |
| --- | --- | --- | --- |
| `0x00` | uint32 little-endian | 地图宽度，单位是 cell | 已确认 |
| `0x04` | uint32 little-endian | 地图高度，单位是 cell | 已确认 |
| `0x08` | char[8] | 魔数/版本字符串 `Hello1.0` | 已确认 |
| `0x10` | record[width * height] | cell 记录数组，每条 16 字节 | 已确认 |

文件大小公式：

`16 + width * height * 16`

例如 `stage11.m`：宽 42，高 168，总 cell 数 7056，文件大小为 `16 + 42 * 168 * 16 = 112912` 字节。

### cell 记录结构

每个 cell 固定 16 字节，小端序。当前字段如下：

| cell 内偏移 | 类型 | 暂定字段名 | 当前理解 | 状态 |
| --- | --- | --- | --- | --- |
| `+0x00` | int16 | `acwx` | 地表基础层图块索引；通常每格都有非负值 | 已确认 |
| `+0x02` | int16 | `acwy` | 地表叠加/边缘/过渡层索引；`-1` 表示无 | 已确认 |
| `+0x04` | int16 | `acwz` | 物件/建筑/树木/城池部件索引；`-1` 表示无 | 已确认 |
| `+0x06` | int16 | `word06` | 目前样本中多为 0 | 已确认存在，语义待定 |
| `+0x08` | uint8 | `byte08` | 稀有非零辅助标志，可能与特殊地形/对象相关 | 待验证 |
| `+0x09` | uint8 | `byte09` | 常见 0/1 标志；是通行/地形/占位候选字段 | 高优先级待验证 |
| `+0x0a` | uint8 | `byte10` | 对象/地形类别候选；EXE 中存在对 `0x65..0x6e` 的比较 | 高置信推断 |
| `+0x0b` | uint8 | `byte11` | 子图块/footprint 子索引候选；部分关卡里常以 36 格为一组出现 | 高置信推断 |
| `+0x0c` | uint8 | `byte12` | 大多为 0 | 语义待定 |
| `+0x0d` | uint8 | `final_palette` | 最终渲染/小图/缓存相关调色板字节候选 | 高置信推断 |
| `+0x0e` | uint8 | `byte14` | 目前样本中多为 0 | 语义待定 |
| `+0x0f` | uint8 | `byte15` | 目前样本中多为 0 | 语义待定 |

因此编辑器不应该只保存三层数字。即使当前 UI 先开放 `acwx/acwy/acwz`，内部数据模型也必须保留完整 16 字节记录，否则保存后会丢失通行、对象 footprint、缓存或其他逻辑字段。

## `acwx/acwy/acwz` 如何构成画面

### `acwx`：基础地形层

`acwx` 是每个 cell 的基础地形图块，几乎所有 cell 都有非负索引。它从 `kingdom.cel` 的 `acwx` 资源组取图，绘制为 40x20 的等距菱形地块。

当前从 EXE 和图像对齐结果确认的世界坐标到屏幕坐标为：

`screen_x = col * 40 + (row 为奇数 ? 20 : 0)`

`screen_y = row * 10`

这意味着地图不是一个传统完整菱形画布，而是按奇偶行错位的 staggered isometric 网格铺开。

### `acwy`：地表叠加/过渡层

`acwy` 同样使用 40x20 菱形图块。它通常不是每格都有，`-1` 表示该格没有叠加层。视觉上它更像道路、水岸、边缘、地形过渡或附加地表细节。

### `acwz`：对象/建筑层

`acwz` 是可高出地表的对象层，包含树、建筑、城池部件等。`acwz` 图块不是固定 40x20，而是带有独立头部：

| 偏移 | 类型 | 含义 | 状态 |
| --- | --- | --- | --- |
| `+0x00` | uint32 | x anchor | 已确认用于绘制偏移 |
| `+0x04` | uint32 | y anchor | 已确认用于绘制偏移 |
| `+0x08` | uint32 | width | 已确认 |
| `+0x0c` | uint32 | height | 已确认 |
| `+0x10` | uint32 | pointer/unknown | 存在，语义待定 |
| `+0x14` | bytes | 像素数据 | 已确认 |

EXE 中的对象绘制逻辑等价于：

`dest_x = screen_x - x_anchor`

`dest_y = screen_y - y_anchor`

当前复原渲染器为了和实际游戏截图对齐，使用地块中心作为对象锚点：

`dest_x = tile_screen_x + 20 - x_anchor`

`dest_y = tile_screen_y + 20 - y_anchor`

这解释了为什么单纯把 `acwz` 垂直居中会“看起来接近”，但并不是完整规则。真正规则必须同时考虑 anchor、footprint、以及绘制顺序。

## 旁路文件与“城池 id”等语义数据

`.m` 的 `acwz` 可以画出城池图形，但图形不等于城池实体。城池 id、所属势力、可进入状态、驻军、事件触发等语义信息，理论上不应该只靠 `acwz` 保存；从文件结构看，它们更可能在以下同名文件中。

### `.stg`：关卡/剧本/城池语义候选

观察到 `.stg` 是变长文件。多个样本开头有稳定 dword：第一个常见为 `1`，第二个常见为 `76`，个别如 `stage00` 为 `72`。文件中还能看到文本内容。

当前推断：`.stg` 很可能保存关卡级配置，例如剧本数据、城池列表、势力/阵营、AI 或初始状态。是否直接保存“城池 id -> 地图坐标/归属”的表，还需要继续对照城池位置和 EXE 访问逻辑确认。

进一步确认：按 `cp950/big5` 解码，`0x08` 附近可以直接读出剧本标题。例如 `stage00.stg` 是“`三國霸業`”，`stage11.stg` 是“`練習一劉備出發`”，`stage20.stg` 是“`為圖長久取四郡`”。另外 `0x24/0x28` 两个 dword 在大量样本里分别呈现 `184/187`、`206/212` 这类年份范围，极像剧本起止年份候选字段。

### `.spr`：单位/士兵数据

`.spr` 文件头为 ASCII：

`Soldier Data`

其后出现稳定的 dword，例如 `180`、`36`，并跟随变长记录。当前高置信推断它保存单位、士兵、武将或地图上动态角色的初始数据。

这类数据不应该写入 `.m` cell；地图编辑器后续需要单独做“单位/对象实体层”编辑。

### `.dor`：门/入口/通道数据

`.dor` 文件头为 ASCII：

`Door    Data`

其后常见 dword `56` 或 `60`。从命名和结构推断，它可能保存城门、入口、通道、关隘或传送/进城点数据。

如果城池有“可进入点”“门是否开启”“入口坐标”等信息，`.dor` 是优先调查对象。

### `.evt`：事件/脚本数据

`.evt` 是事件/脚本文件。第二个 dword 在样本中稳定为 `72`，并且 `stage01.evt` 到 `stage04.evt` 可以被精确解释成 `8` 字节头加 `251` 条 `72` 字节记录；其他关卡则常见“主记录表 + 小尾巴”结构。当前已提取到的 ASCII 命令词包括 `talk`、`VIEW`、`MAP`、`MAPALL`、`MOVE`、`OPEN`、`TIME`、`TIMEOVER`，进一步支持它是脚本命令层。

地图编辑器不应把事件触发直接等同于 `.m` 的图块。后续应把 `.evt` 做成独立事件层，并通过坐标或对象 id 与地图关联。

## `.s` / `.x` 与小地图、通行性的关系

`.s` 和 `.x` 文件都是 25600 字节，也就是 `160 * 160` 的 1 字节栅格。它们不像 `.m` 一样保存变尺寸 cell 表，而是固定尺寸逻辑图。

当前已知事实：

- `.s`：多数关卡存在，大小固定 25600 字节。
- `.x`：多数关卡存在，大小固定 25600 字节。
- 它们更像游戏内部用的固定分辨率地图缓存、缩略图、通行/寻路表、战争迷雾或地形分类表。
- 不能确认二者谁是小地图、谁是通行表；需要从 `Emperor.exe` 对 `.s`/`.x` 的读写逻辑继续验证。

编辑器设计上要预留一个“派生缓存重建”步骤：修改 `.m` 后，不能只更新大地图画面，也要能重建或同步 `.s`/`.x`，否则游戏里可能出现缩略图、寻路或通行信息与画面不一致。


### 当前相关性结论（2026-06-24）

使用 `tools/analyze_stage_sidecars.py` 对 `stage01`、`stage11`、`stage20`、`stage29` 做统计后，当前可以再往前确认几件事：

- `.s` 和 `.x` 都是固定 25600 字节，但二者并不是完全不同的两份数据；四个样本的逐字节相同率分别为 86.31%、80.94%、79.57%、72.71%。
- `.x` 与 `.m` 的 `final_palette` 唯一值集合重叠度 consistently 高于 `.s`。例如 `stage11` 中，`.s` 与 `final_palette` 重叠 62 个唯一值，而 `.x` 重叠 76 个。
- 这说明 `.s` / `.x` 更像同一逻辑平面的两个派生栅格、缓存或不同阶段结果，而不像彼此无关的原始资源文件；其中 `.x` 目前看起来更接近最终缩略图或颜色结果。
- 采样关卡中，`.s` 从未出现“某像素非 `240`，但 `.x` 却是 `240`”的情况；反过来 `.x` 还会比 `.s` 多出 `24` 到 `190` 个非 `240` 像素。结合二者大部分非 `240` 像素值仍然相同，这更像“`.x` 是较完整的颜色缓存，`.s` 是更保守的遮罩/裁剪伴随层”。
- `Emperor.exe` 的字符串上下文中，`.s`、`.x`、`.m` 紧邻出现；而 `.evt`、`.dor`、`.spr` 则与 `stage01.evt`、`stage01.dor`、`castle.magic.txt`、`soldier.txt`、`general.txt` 相邻，进一步支持“地图主表”和“旁路语义文件”分工。

## 通行性数据的候选位置

目前不能断言“是否可通行”只在一个字段中。候选位置按优先级如下：

| 候选 | 证据 | 风险 |
| --- | --- | --- |
| `.m` 的 `byte09` | 常见 0/1 分布，像布尔标志；`stage11` 中 1 的数量为 1688/7056 | 也可能是渲染或地形类别标志 |
| `.m` 的 `byte08/byte10/byte11` | 有特殊值和成组分布，`byte10` 被 EXE 与 `0x65..0x6e` 比较 | 可能表示对象 footprint、地形类别或城池占位，不一定直接等于通行 |
| `.s`/`.x` | 固定 160x160 栅格非常适合存通行/寻路/小地图缓存 | 目前未确认 EXE 使用语义 |
| `.dor` | 门/入口类数据通常会影响通行 | 可能只影响特殊入口，不负责普通地形 |

下一步验证方式：

1. 在 EXE 中定位 `.s`/`.x` 文件名引用和加载函数。
2. 找到寻路、鼠标点击、单位移动或建筑占位相关代码，看它读取哪一份数据。
3. 用已知河流、山地、城墙、道路位置对照 `byte09`、`byte10`、`.s`、`.x` 的空间分布。
4. 在复制关卡上做最小改动，进入游戏验证单位是否能通过。

| 字段 | 观察结果 |

| `acwy` | 392 种，`-1` 出现 5354 次 |
| `acwz` | 134 种，`-1` 出现 6312 次 |
| `word06` | 全部为 0 |
| `byte08` | `0` 出现 6992 次，`138` 出现 64 次 |
| `byte09` | `0` 出现 5368 次，`1` 出现 1688 次 |

| `final_palette` | 77 种值 |

### `stage20.m`

`stage20` 中 `byte11` 出现 0..12 等值，多个值以 36 cell 为一组出现。这很像大型对象的 footprint 或子块编号，而不是普通随机标志。

### `stage01.m`

`stage01` 中 `byte08`、`byte10`、`byte11` 的取值更丰富，说明这些字段在复杂地图/城池/特殊地形中更活跃。它是继续研究对象层和 footprint 的重要样本。

## 编辑器设计影响

1. `.m` 编辑器内部必须保留完整 16 字节 cell 记录。UI 可以先只显示 `acwx/acwy/acwz`，但保存时要原样带回未编辑字段。
2. 资源面板需要同时展示 `acwx/acwy/acwz` 的图形预览。选择资源时，写入的是索引；画面重绘时，从 `kingdom.cel` 对应资源组取图。
3. 城池、单位、门、事件不应作为 `acwz` 的附属字符串硬塞进 `.m`。后续应分别解析 `.stg/.spr/.dor/.evt`，做成独立可见图层。
4. 通行性编辑要等字段确认后再开放写入。短期可以做“只读 overlay”：把疑似通行字段叠加显示，帮助人工验证。
5. 修改 `.m` 后，还需要设计 `.s`/`.x` 的同步策略。最保守方案是先导出补丁和新 `.m`，暂不改 `.s`/`.x`；等 EXE 逻辑确认后再自动重建。
