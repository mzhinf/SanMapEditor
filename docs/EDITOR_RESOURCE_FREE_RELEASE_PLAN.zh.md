# 地图编辑器无资源发布包实施计划

更新日期：2026-07-16

## 一、文档用途

本文档用于将“地图编辑器发布包不包含任何游戏资源”需求交接给独立开发任务。后续对话应以本文档作为需求边界、实施顺序和验收依据，并在实现过程中同步维护本文档及相关现行文档。

本文档现同时作为已实施方案的验收基线。代码发布包、运行时导入、临时会话、数据回退清理、四态启动器、资源审计和现行文档同步均已实现；不得恢复旧的构建时关卡 Bundle 发布方式。

### 1.1 当前实施状态

- 阶段 1 至 9 已按顺序独立实施和提交。
- 正式构建不接受 `--stage`，不读取 `data/game` 或 `data/text`。
- 发布目录和 ZIP 使用五文件精确白名单，并审计 PyInstaller TOC。
- 运行时只使用用户显式输入，在系统临时目录事务生成会话。
- 真实 `stage01`、CP950、`stage.ini`、Sidecar 和完整自动测试已经通过。
- 正式 EXE `--check`、目录审计、ZIP 审计和哈希清单已经通过。
- 最终完整回归 124 项通过，1 项按样本条件跳过。
- 最终 ZIP 位于 `dist/resource-free-final/SanMapEditor-2026-07-16.zip`，SHA-256 为 `7051aa2be6e9c4c40a75955874e84b171c807eaefe3549e08c45bd1e7ce839c5`。

仍需由发布维护者在合法游戏副本中执行最终人工兼容性验证，并对 `SAN_RGB_PALETTE` 的公开发布边界完成法律确认。这两项外部门禁不允许被自动测试或资源扫描替代。

## 二、需求原文与目标

原始需求：

> 鉴于版权问题，导出的地图编辑器不包含任何资源文件。发布包内没有 `stage` 文件夹，所有地图、头像和资源贴图都由用户导入。

目标是把发布方式从“构建时携带一个可直接编辑的关卡”改为“只发布编辑器程序，运行时读取用户自行准备的合法游戏文件”。

最终发布 ZIP 不得包含：

- 原始游戏文件；
- 从原始游戏文件复制、转换或渲染得到的派生资源；
- 构建机器上的测试关卡、资源目录或文本母表；
- 能够还原具体游戏地图、头像、角色、城池或贴图内容的数据。

本项目不提供法律意见。移除资源可以降低再分发游戏素材的风险，但不能自动证明源码、名称、格式研究结论或其他内容已经获得公开发布许可。

## 三、范围定义

### 3.1 发布包内允许存在

推荐的最小发布结构如下：

```text
三国霸业地图编辑器/
├─ SanMapEditor.exe
├─ editor-data/
│  ├─ index.html
│  └─ release-info.json
├─ 使用说明.txt
└─ 编辑器使用指南.md
```

允许包含编辑器源码编译产物、HTML/CSS/JavaScript、格式定义、版本信息、使用文档以及程序自身的 UI 图标和配色。

### 3.2 发布包内禁止存在

至少禁止以下路径或文件：

```text
editor-data/stage*/
*.m
*.dor
*.stg
*.s
*.x
stage.ini
History.txt
heads.dat
kingdom.cel
kingdom.atr
stage_ini.xlsx
```

游戏资源生成的 `map.png`、`minimap.png`、`heads.png`、`draw_acw*.png`、`resources_acw*.png`、`stage.json` 和 `resources.json` 同样不得进入发布包。普通 UI 图片是否允许，应由发布白名单按具体文件确认，不能仅按扩展名一刀切。

### 3.3 “没有 stage 文件夹”的解释

强制要求是最终发布目录和 ZIP 中没有 `stageXX` 文件夹。

推荐实现允许程序在用户导入资源后，于系统临时目录创建本次会话使用的运行时数据，例如：

```text
%TEMP%/SanMapEditor/<会话编号>/
```

这些数据只能来自用户本机导入的文件，不能写回安装目录，退出程序后应自动清理。若连运行时临时目录也禁止创建，则必须改为纯浏览器内存解析，工作量和回归风险会显著增加，详见第六节。

## 四、当前实现与目标之间的差距

当前发布链路会在构建阶段读取 `data/game`、`data/text` 或对应环境变量目录，并生成完整关卡 Bundle。以下功能仍隐式依赖 Bundle：

1. `editor_app.html` 启动时请求 `stage.json`、`resources.json`、关卡页面、地图图片和资源图集。
2. 地图底图、小地图、三层资源库和主画布重建依赖预生成 PNG 图集。
3. 武将头像显示依赖从 `heads.dat` 预生成的 `heads.png`。
4. 导入 `.m` 时会继承已打包关卡中的字段元数据、调色板、场景模型和公共模型。
5. `stage.ini` 编辑依赖预生成的 Patch 模型，并间接依赖 `data/text` 的关联信息。
6. 新增中文城池和武将名称依赖 Big5/CP950 字符映射；只分析当前关卡字符不能覆盖全部新增名称。
7. `.s/.x` 导出可能从 `stage.json` 读取打包时保存的尾区数据；缺失时存在零填充回退。
8. 导出逻辑可能在用户未导入对应文件时请求发布包中的原始参考文件。
9. 桌面启动器要求 `index.json` 和关卡页面存在，尚不支持空项目启动。

因此，直接删除现有 `editor-data/stageXX` 会导致启动、渲染、头像、INI 和部分导出功能失效。

## 五、推荐架构

采用“代码发布包 + 启动器运行时导入 + 临时会话 Bundle”架构。

```mermaid
flowchart LR
    A["无资源发布 ZIP"] --> B["启动 SanMapEditor.exe"]
    B --> C["选择 stageXX.m 或游戏目录"]
    C --> D["识别并校验同组文件"]
    D --> E["在系统临时目录生成会话数据"]
    E --> F["启动本地 HTTP 服务"]
    F --> G["浏览器编辑地图"]
    G --> H["导出用户修改后的完整项目"]
    B --> I["退出时清理临时会话"]
```

采用该架构的原因：

- 可以复用现有 Python 的 `.m/.dor/.stg`、`kingdom.cel`、`heads.dat` 和 INI 解析能力；
- 避免在 JavaScript 中再次实现一套容易分叉的二进制格式解析器；
- 保留当前已经验证过的渲染、场景关联和导出链路；
- 发布 ZIP 内没有游戏资源，运行时数据只来自用户选择的文件；
- 可以用显式会话生命周期管理临时文件和错误恢复。

## 六、备选架构

严格的纯浏览器方案不创建任何临时 `stage` 目录：根页面空状态启动，用户在浏览器内一次性导入全部文件，JavaScript 直接解析二进制并在 Canvas 或内存图像中生成资源。

该方案还需要实现：

- JavaScript 版 `kingdom.cel` 解析器和动态资源图集；
- JavaScript 版 `heads.dat` 头像解析与调色；
- 完整 CP950 编码器或本地后端编码接口；
- `stage.ini` 布局推导及新增行写回；
- `.s/.x` 尾区保存、校验与导出；
- 大地图资源缓存、内存释放和浏览器容量保护。

除非明确禁止运行时临时目录，否则不推荐优先选择该方案。

## 七、输入文件与导入规则

用户可选择 `stageXX.m` 或游戏资源目录。选择 `.m` 时，应以文件名中的 `stageXX` 作为场景编号，在同目录或用户明确选择的资源目录中匹配：

| 文件 | 建议级别 | 用途 |
| --- | --- | --- |
| `stageXX.m` | 必需 | 地图尺寸和 Cell 数据 |
| `kingdom.cel` | 必需 | `acwx/acwy/acwz` 地图资源 |
| `kingdom.atr` | 待核实 | 资源属性和相关分析 |
| `stageXX.dor` | 场景管理必需 | 据点城门记录 |
| `stageXX.stg` | 场景管理必需 | 势力、据点、武将和士兵 |
| `stageXX.s`、`stageXX.x` | 完整导出必需 | 小地图 Sidecar 及尾区 |
| `stage.ini` | 新增对象必需 | 城池和武将母表 |
| `History.txt` | 武将管理必需 | 历史武将与未加入剧本记录 |
| `heads.dat` | 头像显示必需 | 武将头像像素数据 |
| 调色板来源 | 渲染必需 | 地图、资源、小地图和头像颜色 |

导入必须具备以下行为：

1. 显示已识别、缺失、重复和场景编号不匹配的文件。
2. 关键文件不完整时不得静默使用程序内测试数据或构建机器数据。
3. 导入失败必须保持原会话不变，不能留下半更新状态。
4. 不允许根据固定绝对路径读取游戏安装目录。
5. 用户可以重新选择资源目录，并明确决定是否替换当前会话。

## 八、调色板版权边界

实施前必须在以下策略中选择一种：

### 策略 A：仅禁止资源文件

保留源码中的 `SAN_RGB_PALETTE` 常量。该策略改动较小，但需要单独确认该常量是否属于允许发布的程序研究数据。

### 策略 B：禁止游戏衍生数据

发布包不携带 `SAN_RGB_PALETTE`，运行时要求用户导入合法调色板来源，或从已经确认格式的用户游戏文件中提取。当前项目尚未确认权威调色板文件和提取规则，实施前需要补充研究和测试。

编辑器自身的标记颜色、覆盖层颜色、按钮颜色和其他 UI 配色不属于地图资源调色板，可以保留。

## 十二、实施阶段

1. **确认边界**：确定是否允许运行时临时目录，以及采用哪种调色板策略。
2. **建立资源审计**：先写发布白名单、禁用文件扫描和干净环境构建测试。
3. **解耦发布构建**：让发布命令在没有任何游戏数据时生成代码包。
4. **实现运行时会话**：完成用户文件识别、校验、临时目录和清理。
5. **迁移资源生成**：地图、图集、小地图和头像改为导入后生成。
6. **解除数据回退**：处理 INI、Big5/CP950、Sidecar 和原始参考文件依赖。
7. **适配编辑器状态**：增加空项目、导入中、导入失败和已加载状态。
8. **完整回归验证**：执行自动测试、Stage 样本测试和游戏副本测试。
9. **更新文档并发布**：同步全部现行文档，生成并审计最终 ZIP。

每个阶段应独立提交。不得先删除现有 Bundle 资源再补功能；应先建立新运行链路和测试，最后移除旧发布路径。

## 十三、验收标准

全部满足以下条件后，需求才算完成：

1. 发布构建不读取 `data/game`、`data/text` 或其他本机游戏资源目录。
2. 最终 ZIP 和 EXE 经审计不包含原始游戏文件及其派生资源。
3. 最终发布目录没有 `stageXX` 文件夹。
4. 编辑器无资源启动时正常显示导入流程，不出现资源请求错误。
5. 用户导入合法完整项目后，地图、资源、小地图、头像和管理面板正常工作。
6. 所有编辑和导出能力不依赖发布包内隐藏参考文件。
7. 新增城池与武将能够正确写入所有关联文件，中文名称编码正确。
8. 临时会话只使用用户导入资源，并在退出后完成清理。
9. 自动测试、发布资源审计、Stage 集成测试和游戏副本验证全部通过。
10. README、用户指南、打包链路、NOTICE 和 GitHub 发布清单均与新行为一致。

## 十四、实施采用的决策

本次实施采用以下决定：

1. 允许在 `%TEMP%/SanMapEditor` 创建带专用标记并自动清理的临时会话。
2. 本次按“禁止资源文件”实施，保留源码 `SAN_RGB_PALETTE`；其公开发布权利仍是单独人工法律门禁。

若后续边界升级为“禁止任何游戏衍生数据”，必须先研究并实现从用户合法文件提取权威调色板的路径，再重新执行全套构建、渲染和游戏兼容性验证。
