# Sangokushi Hegemony / Emperor Map Format Notes

This document records the recovered map/resource structure observed from `Emperor.exe` and the game asset directory.

## Executable

`Emperor.exe` is a PE32 i386 executable with four sections: `.text`, `.rdata`, `.data`, `.rsrc`.
The resource directory is present, but the map data is primarily external. Static strings in the executable reference:

- `stage00.stg` through `stage46.stg`
- `stage01.evt`, `stage01.dor`, `.spr`
- `kingdom.cel`, `graphics.dat`, `windows.dat`, `selects.dat`, `heads.dat`
- editor-like filters: `Text Files (*.STG)` / `ASCII Files (*.MAP)`

## Stage Files

Each stage is split across several files:

- `stageNN.m`: pre-rendered indexed map image/layer.
- `stageNN.s`: 160x160 byte grid, likely base terrain/tile layer.
- `stageNN.x`: 160x160 byte grid, likely edited/overlay terrain or passability-related terrain variant.
- `stageNN.stg`: scenario metadata; starts with record-like little-endian integers and strings.
- `stageNN.spr`: starts with ASCII `Soldier Data`; unit/object records.
- `stageNN.dor`: starts with ASCII `Door    Data`; gate/door/castle access records.
- `stageNN.evt`: event/script data; second dword often `0x48`, suggesting 72-byte records in at least part of the file.

## `stageNN.m`

Recovered structure:

```text
uint32 width
uint32 height
char[8] magic = "Hello1.0"
record[width * height]
```

Each record is 16 bytes. The byte at offset 13 within each record is a palette index. Rendering that byte as an 8-bit indexed bitmap with the game's BMP palette reconstructs the visible stage map.

Validated examples:

- `stage01.m`: `336 * 1072 * 16 + 16 = 5,763,088` bytes.
- `stage20.m`: `112 * 356 * 16 + 16 = 637,968` bytes.
- `stage39.m`: `126 * 400 * 16 + 16 = 806,416` bytes.

Other record fields appear to contain source tile/object references and flags. Common interpretation candidates:

- bytes 0..1: little-endian coordinate/reference value.
- bytes 2..5: secondary references, often `0xffff` when absent.
- bytes 8..11: flags/object/cell references.
- byte 13: final rendered palette index.

## `.s` and `.x` Grids

All observed `.s` files are exactly 25,600 bytes, matching a 160x160 byte grid. Most `.x` files are also 25,600 bytes. The byte values match palette/tile indices used in `.m`, so these are editor-critical logical map layers rather than compressed images.

## DAT Image Containers

`Graphics.dat`, `Selects.dat`, `windows.dat`, and `heads.dat` are indexed image containers.

The file begins with a table of little-endian uint32 offsets. The first dword is the first data offset, so the maximum table slots are `first_offset / 4`. Valid offsets are strictly increasing until a zero or non-increasing value.

Each chunk uses:

```text
uint16 width
uint16 height
uint8 pixels[width * height]
```

The pixels use the same 256-color game palette. Exported counts from current parser:

- `Graphics.dat`: 82 images.
- `Selects.dat`: 32 images.
- `windows.dat`: 332 images.
- `heads.dat`: 413 images.

## Update 2026-06-23: Kingdom CEL/ATR Recovery

`kingdom.cel` and `kingdom.atr` are now partially decoded enough for editor work.

### `kingdom.atr`

The ATR file is a sparse stream of 6-byte records mixed with gaps:

```text
char[4] code    // `acwx`, `acwy`, or `acwz`
uint8 attr_lo
uint8 attr_hi   // commonly matches rendered terrain/object color class
```

Current parser finds 6,344 ATR records:

- `acwx`: 2,052 records
- `acwy`: 2,223 records
- `acwz`: 2,069 records

### `kingdom.cel`

CEL contains image chunks using the same 8-bit palette.

`acwx` and `acwy` are fixed 20x20 chunks:

```text
char[4] code    // `acwx` or `acwy`
uint8 pixels[20 * 20]
```

`acwz` is variable-size and stores larger isometric object/building slices:

```text
char[4] code = `acwz`
uint32 x0_or_source_x
uint32 y0_or_source_y
uint32 width
uint32 height
uint32 pointer_or_unknown
uint8 pixels[width * height]
```

Current parser finds 8,320 CEL chunks:

- `acwx`: 1,944 fixed 20x20 chunks
- `acwy`: 2,162 fixed 20x20 chunks
- `acwz`: 4,214 variable-size chunks

The ATR and CEL counts do not match exactly. This likely means ATR contains placeholder/no-image attributes, while CEL stores actual drawable chunks. For map reconstruction, use ATR order for `.m` field metadata and CEL chunks for drawable resources where available.

### `.m` Record Relationship

Each `stageNN.m` 16-byte record is not just a final pixel. It also stores tile/object references. The first signed 16-bit field (`record[0:2]`) behaves as a primary tile index into ATR order. Rendering `ATR[tile_id].attr_hi` produces a recognizable base/contour map and often matches the final rendered byte at `record[13]`.

Examples of field0-to-final-byte direct match rates:

- `stage01`: 52.2%
- `stage11`: 51.2%
- `stage20`: 23.6%
- `stage39`: 48.7%

Lower rates likely indicate more overlay/object contribution from the other signed 16-bit fields and flags in the 16-byte record.

### Revised Classification

- `derived/maps/*_m.png`: large pre-rendered map/cache layer, useful but not the raw editable tile map by itself.
- `derived/grids/*_s.png` and `*_x.png`: likely minimap/screen-cache layers, not final game map data. Their bottom rows are largely shared UI/cache content.
- `kingdom.cel/.atr`: actual terrain/object image resources and tile attributes.
- `stageNN.m`: best current source for reconstructing per-pixel tile/object references; field0 is the primary terrain/object index, other fields need more decoding.

### CEL Tile Composition Update 2026-06-23

`acwx` chunks appear to be ordered as terrain meta-tiles. The strongest current visual reconstruction is 36 consecutive 20x20 chunks arranged as a 6x6 block, yielding 120x120 terrain patches. Exported preview: `derived/kingdom/acwx_meta_6x6_sheet.png`.

`acwy` chunks do not form coherent terrain patches under 3x3, 4x4, or 6x6 consecutive grouping. The data is likely an overlay, mask, transition, or secondary terrain/object layer rather than the same base terrain tile sequence as `acwx`.

`acwz` chunks are vertical strips for larger isometric city/object sprites. Consecutive ranges form complete buildings when placed left-to-right. The best current visual alignment is vertical centering of each strip within the group's maximum height; bottom alignment works but visually pushes all strips to a shared baseline and looks less faithful for city sprites. The first decoded representative groups are:

```text
0:12, 12:22, 22:33, 33:45, 45:55, 55:67, 67:77, 77:87,
87:98, 98:109, 109:120, 120:131, 131:145, 145:154, 154:164
```

Exported previews: `derived/kingdom/acwz_align_compare.png` and `derived/kingdom/acwz_stitched_city_center.png`.

## Update 2026-06-23 Emperor.exe ACWZ Draw Logic
- Installed and used Capstone to disassemble `Emperor.exe` locally.
- `kingdom.cel` loader is at file offset `0x27480` / VA `0x427480`.
- `acwz` draw branch is around file offset `0x19f0b` / VA `0x419f0b`.
- The draw branch reads the stage/map record word at `[record + 4]` as the `acwz` logical index, then looks up `0x473654[index]`.
- The decoded `acwz` header is used as anchor metadata:
  - `+0`: x anchor / x offset
  - `+4`: y anchor / y offset
  - `+8`: width
  - `+0x0c`: height
  - `+0x10`: temporary pixel buffer during load, freed after creating the surface
- Draw formula recovered from the exe: `dest_x = screen_x - header0`, `dest_y = screen_y - header1`, then width/height are clipped before DirectDraw Blt.
- For all 4,214 decoded `acwz` chunks, `header1 == height`, so every strip is bottom-anchored to its owning map cell baseline.
- Surface pointers are stored separately in a table around `0x4871fc`, with logical-to-surface short mapping at `0x47eaa8`.
- Generated `derived/kingdom/acwz_stitched_city_exe_anchor_x20.png`, a closer preview using `cell_x += 20`, `dest_x = cell_x - header0`, `dest_y = baseline - header1`, and transparent color 0 compositing.
- Remaining issue: a full city sprite is not a single image; it is multiple map cells referencing consecutive `acwz` strips. Exact vertical micro-alignment requires recovering each strip's real map cell `screen_y` from stage placement logic.

  - `acwy` block: 8,640 logical slots, 2,223 ATR-present entries.
  - `acwz` block: 17,280 logical slots, 2,069 ATR-present entries.
- `.m` record schema now has a stable first-pass interpretation:
  - `+0x00 int16`: `acwx` base terrain tile index; always present in observed stages.
  - `+0x02 int16`: `acwy` overlay/transition index; `-1` when absent.
  - `+0x04 int16`: `acwz` object/building/footprint index; `-1` when absent. Some non-negative entries point to empty logical slots, likely object footprint/occupancy cells.
  - `+0x06 int16`: observed zero in exported `.m` files.
  - `+0x08/+0x09`: auxiliary bytes; common values suggest terrain/object flags, not fully named yet.
  - `+0x0a`: variant/object byte. `Emperor.exe` compares this against `0x65..0x6e`, matching common values 101..110.
  - `+0x0b`: small subindex/group byte; often appears in 36-cell groups, likely footprint/subtile index.
  - `+0x0d`: final rendered palette index/cache byte.
  - `+0x0c/+0x0e/+0x0f`: observed zero in saved `.m` files; runtime renderer has mode/flag checks against related in-memory offsets.
- Added `tools/export_m_layers.py` to export editor-oriented layers under `derived/m_layers/`:
  - `acwx.i16`, `acwy.i16`, `acwz.i16`
  - `byte08.bin`, `byte09.bin`, `byte10.bin`, `byte11.bin`, `final_palette.bin`
  - `*_attr.png` previews and `meta.json` per stage
  - global `m_record_summary.tsv` and `kingdom_blocks.json`
- Visual layer check on `stage01` confirms: `acwx_attr` is the full base terrain; `acwy_attr` is contour/shore/road/transition overlay; `acwz_attr` is sparse object/city footprint placement.
