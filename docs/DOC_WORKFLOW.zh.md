# 文档维护约定

这份文档定义本项目的强制文档流程。目标只有一个：任何一次有效进展，都必须同步留下可复用文字，而不是只存在聊天上下文里。

## 1. 文档分工

| 文件 | 负责内容 | 何时更新 |
| --- | --- | --- |
| `README.md` | 项目入口、脚本用法、当前能力边界 | 脚本入口、参数、产物路径变化时 |
| `docs/FORMAT_NOTES.zh.md` | 二进制格式、字段语义、证据级别 | 对格式理解有新增或修正时 |
| `task_plan.md` | 阶段目标、完成状态、下一步计划 | 阶段切换或优先级变化时 |
| `findings.md` | 当前稳定结论 | 新结论成立或旧结论废弃时 |
| `progress.md` | 本次做了什么、如何验证 | 每次实际修改或验证后 |

## 2. 每次工作最少要做的事

1. 改动前先看 `task_plan.md`，确认当前阶段与边界。
2. 形成有效结果后更新 `progress.md`。
3. 如果结论影响格式理解，更新 `docs/FORMAT_NOTES.zh.md` 与 `findings.md`。
4. 如果脚本入口、参数或产物路径变化，更新 `README.md`。

## 3. 提交前检查表

每次 git 提交前至少检查：

1. 新脚本是否已经写入 `README.md`
2. 新的格式结论是否已经写入 `docs/FORMAT_NOTES.zh.md`
3. `task_plan.md` 是否反映当前阶段
4. `progress.md` 是否写清验证方式
5. 文档里是否还残留失效命令、错误路径、乱码或重复段落

只要其中任一项答案是否定，就先补文档，再提交代码。

## 4. 文档质量标准

允许保留“未知”，但不允许保留“已经失效的旧说法”。

文档最低标准：

- 命令可以直接执行
- 路径真实存在
- 结论标明证据级别
- 明确哪些名字是分析标签，哪些是源文件事实
- 全部中文文档统一保存为 UTF-8

## 5. 明确禁止的做法

以下行为视为不合格：

- 只改脚本，不改文档
- 只在聊天里说结论，不落文件
- 明知文档过期，仍继续堆叠新内容
- 保留乱码，寄希望于“以后再修”

## 6. 当前执行方式

从当前版本开始，以上规则作为仓库默认约束执行。如果后续发现某类信息总是容易漏写，就新增对应文档或检查项，而不是继续依赖记忆维护。
