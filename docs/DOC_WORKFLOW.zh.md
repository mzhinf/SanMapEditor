# 文档维护约定

这份文档是本项目的“强制文档流程”。
目标只有一个：任何一次有效进展，都必须同步留下可复用文字，而不是只留在对话里。

## 1. 文档分工

| 文件 | 负责内容 | 何时更新 |
| --- | --- | --- |
| [README.md](/H:/Workstation/san/README.md) | 项目入口、脚本用法、当前能力边界 | 脚本入口、使用方式、产物路径变化时 |
| [docs/FORMAT_NOTES.zh.md](/H:/Workstation/san/docs/FORMAT_NOTES.zh.md) | 二进制格式、字段语义、证据级别 | 对格式理解有新增或修正时 |
| [task_plan.md](/H:/Workstation/san/task_plan.md) | 阶段目标、完成状态、下一步 | 阶段切换、优先级变化时 |
| [findings.md](/H:/Workstation/san/findings.md) | 新发现、新结论、关键判断依据 | 每出现新的稳定结论时 |
| [progress.md](/H:/Workstation/san/progress.md) | 本次做了什么、如何验证、遇到什么问题 | 每次实际修改或验证后 |

## 2. 每次会话的最小动作

每次工作至少完成下面 4 件事：

1. 做改动前，先读 `task_plan.md`。
2. 做出有效结果后，写 `progress.md`。
3. 如果结论影响格式理解，写 `findings.md` 和 `docs/FORMAT_NOTES.zh.md`。
4. 如果脚本入口、文件路径、操作步骤变了，写 `README.md`。

## 3. 提交前检查表

每次 git 提交前，按下面顺序检查：

1. 本次新增脚本是否已写入 `README.md`
2. 本次新增格式结论是否已写入 `docs/FORMAT_NOTES.zh.md`
3. `task_plan.md` 是否反映最新阶段
4. `progress.md` 是否写清楚了验证方式
5. 文档里是否还有失效命令、错误路径、乱码、重复段落

如果其中任意一项答案是否，则先补文档，再提交。

## 4. 文档质量标准

允许保留“未知”，但不允许保留“含糊的旧信息”。

因此文档必须满足：

- 命令能直接执行
- 路径真实存在
- 结论注明“已确认 / 高置信推断 / 待验证”
- 删除已经失效的旧说法，不把冲突版本并列堆着
- 中文文件统一保存为 UTF-8

## 5. 反模式

下面这些行为以后都视为不合格：

- 只改脚本，不改文档
- 只在聊天里说结论，不写到文件
- 明知文档过期，仍继续累加新内容
- 保留乱码和重复段落，指望“以后再收”

## 6. 当前执行方式

本项目从本次开始按此规则执行。
如果后续发现某类信息总是容易漏写，就新增对应文档或检查项，而不是继续靠记忆维护。
