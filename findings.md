# Stage.ini 写入错误与修复结论

## 样本

- 游戏有效文件：`data/game/stage.ini`，75,280 字节，SHA-256 `29584DE26770323A09849D180331D936E9C112F55936D76B08F4F6F6A63663B8`。
- 首次崩溃样本（当时的 `outputs/stage.ini`）：75,880 字节，SHA-256 `A2026A0BFD66F95E0F6F7B661B5A871C5E00CB065BB7AC2833878C36894CA948`。
- 首次块边界修复样本：75,608 字节，SHA-256 `F64737E1A6A0CF01F2B94118132B471E04C2131FF7121718371BF0C1C0BDE3F3`。
- 当前人物 ID 对齐输出：75,836 字节，SHA-256 `2038FDF997EE042D6A35AA63247504AD220D3CA1A5EA34368CAFBF5F438AC540`。
- 文件是含 NUL 的二进制母表，不是文本 INI；新增标题可按 CP950/Big5 正确解码，编码不是本次崩溃根因。

## 真实物理结构

- 文件以 `main_count:u4` 开始，随后是 `main_count` 个 `size:u4 + payload[size]` 主块。有效样本有 277 个主块，`size` 均为 224，主块流结束于 63160。
- 前 272 个主块是普通武将：20 字节标题/类型槽、49 个命名 DWORD、2 个能力镜像 DWORD；加上 size 前缀后每条共 228 字节。
- 偏移 63160 是 `city_count=42`。每座城池由 `size=92` 主块和 `size=0` 次块组成：主块内为 1 个前导 DWORD、20 字节标题/类型槽、17 个命名 DWORD；每座共 100 字节。
- 42 座城池结束于 67364。此前把 67364、67368 的两个 `76` 归入“南海”尾字段是错误的；它们属于城池区之后的下一资源块。
- 旧的“8 字节头 + 224 字节主视图 + 76 字节尾视图”只是一种可字节一致回写的兼容工作簿视图，不是物理记录格式。

## 根因

- `_stage_ini_append_layout` 根据相邻命名数值流反推标题和行长，错误得到 32 字节标题槽，并在最后命名字段后直接插入。
- 武将从 62012 插入会覆盖最后一名普通武将的两个能力镜像；62020 是下一个合法块边界，但人物 ID 278/279 仍不能放在这里。新增武将必须追加到全部原主块结束位置 63160。
- 城池因此从 67360 插入，覆盖最后一座原城池的 `size=0` 次块；正确插入点是 67364，不是初步分析中的 67372。
- 前端又把 228 字节武将行补齐为 448 字节、100 字节城池行补齐为 152 字节，净增 600 字节，并只按伪主步长重算一个计数。

## 修复

- 新增真实块流解析器，输出每个块的 size、payload 与结束偏移。
- 新增武将写入完整 228 字节块，保留标题类型标记，并从“最大武力/最大智力”写入两个尾随镜像。
- 新增城池写入完整 100 字节双块，保留前导值、标题类型标记和零长度次块。
- 主块数和城池数分别在位移后的真实偏移更新，不再执行 224/76 步长补齐。
- 首次修复样本曾从游戏有效母表重建并迁移武将“劉飛羽”和城池“北都”；当前 `outputs/stage.ini` 已是后续新增两名武将和“晉都”的新样本。
