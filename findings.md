# 结论记录

## 项目目标

围绕《三国霸业》建立一条完整链路：

1. 从原始游戏目录恢复真实地图。
2. 建立可编辑、可回写的地图数据模型。
3. 明确 `stage.ini`、`.stg` 与各类 sidecar 的边界。
4. 把所有稳定结论落到脚本和文档中，而不是只留在对话里。

## 当前稳定结论

### 1. `.m` 是地图主表

- 文件头固定 16 字节：`width`、`height`、`Hello1.0`
- 每个 cell 固定 16 字节
- 前 6 字节稳定对应 `acwx`、`acwy`、`acwz`
- 其余字节不能视为噪声，编辑器必须完整保留

### 2. `kingdom.cel/.atr` 是真实地图资源核心

- `acwx`：基础地形
- `acwy`：叠加/过渡
- `acwz`：建筑/物件
- 当前渲染器已经能按真实资源而非占位图绘制地图

### 3. `Emperor.exe` 已经给出关键渲染逻辑

- 地图采用 staggered isometric 网格
- 当前 world-to-screen 变换已经足够贴近 `stage11` 实机画面
- 剩余差异主要集中在 `acwz` 物件的 footprint 与 z-order

### 4. `stage.ini` 是全局二进制母表

- 不是文本 ini
- 已有 JSON / Excel / 二进制互转脚本
- 纯转换工作簿在未修改时能够回写出字节完全一致的新文件

### 5. `stage01.stg` 更像顺序脚本而不是独立实体表

- 当前最稳的读取方式是：势力块 -> 城池块 -> 城内武将/士兵/附属记录
- 大量无文本记录仍然属于有效结构，不能在导出时丢掉
- `family_guess`、`city_92_family` 等名字都是分析标签，不是源文件字段名

### 6. `.stg` 已具备安全工作簿回写能力

- `raw_records.raw_hex` 作为保底数据源
- `city_state` 中已经确认的 `candidate_*` 字段可回写
- `troop_candidates` 目前仍以只读分析为主

### 7. `.evt` 已经能与外部文本资源对上主轮廓

- `.evt` 当前最稳的文件框架是：8 字节头 + 72 字节记录体。
- 全量统计显示 38 个 `.evt` 都能对上对应的 `TalkNN.txt`，可作为事件文本资源池。
- `stage17.txt` 是可读的 CP950 松散脚本原型，能和 `stage17.evt` 中的 `呂翔死`、`徐庶回軍寨` 等标签互相印证。
- `stage01.txt` 不是普通文本，而是误用 `.txt` 后缀的二进制 blob，不能和 `TalkNN.txt` 混为一类。
- 目前最稳定的 ASCII 命令 token 是 `talk`、`VIEW`、`MAP`、`MAPALL`、`MOVE`、`TIME`、`TIMEOVER`。
- `新增流程`、`流程`、`y程` 这类字符串大量来自命令区与文本解码重叠，不能直接当成可靠业务字段名。
- 仅凭“小整数也落在 Talk 段号空间内”还不能认定它就是文本引用；坐标、对象和全局 id 仍需继续拆解。

### 8. `.s/.x` 可以按“有效区 + 尾区”保守重建

- `.s/.x` 都是固定 160x160 的单字节网格。
- 33 个带 `.m/.s/.x` 的关卡中，`.s` 与 `.x` 平均仍有 `0.80094` 的逐字节相同率，说明二者高度相关。
- 把 `.m` 的 `byte13 / minimap_color` 直接缩放成 160x160 仍然不对，但若只看顶部 `128` 行，有效区与 `.s/.x` 的平均匹配率可达 `0.47098 / 0.620744`，其中 `.x` 始终更接近 `.m` 派生结果。
- 跨关卡统计里，第 `128..141` 行和 `150..159` 行在 `.s/.x` 中都出现了 `>= 0.95` 的静态一致度，`159` 行甚至达到 `1.0`，说明底部 `32` 行可以视为公共尾区或缓存区。
- 因此，当前最稳的写回策略已经明确：上 `128` 行由 `.m byte13 / minimap_color` 缩放生成，下 `32` 行直接保留原始 `.s/.x` 副本。
- 现有 `tools/build_minimap_sidecars.py` 已把这条规则脚本化；全量 33 个关卡里，生成结果的尾区匹配率恒为 `1.0`。
- `tools/apply_editor_patch.py` 已经把这条 sidecar 规则接进编辑器 patch 写回链路，能够一次完成 patch -> `.m/.s/.x` 闭环。
- 剩余未解的是：`.x` 为什么总比 `.s` 更接近 `.m`，以及 `Emperor.exe` 是否还会对顶部有效区再做额外调色或标记。

### 9. `.dor` 已能做候选字段与地图关系可视化

- `.dor` 文件头带有 `Door    Data` 魔数，当前样本中可以稳定识别出头部与定长记录区。
- 重新从 `Emperor.exe` 复核后可见：`.spr/.dor/.evt` 的扩展名引用点落在两段紧凑代码簇，而 `.m/.s/.x` 在另一段；这支持 `.dor` 是与 `.m` 并列的关卡 sidecar，而不是其内嵌镜像。
- `Door    Data` 魔数只被两处代码引用，且都出现 `push 0x0c; push "Door    Data"` 头校验模式，进一步说明 `.dor` 走的是独立 sidecar 装载路径。
- `tools/analyze_dor_relationship.py` 已更新为同时输出候选坐标覆盖图与 `Emperor.exe` 证据摘要，报告写入 `derived/dor_relationship/<stage>/dor_relationship.json`。
- 目前最像坐标对的字段组合已经能在部分关卡上落到合理地图范围内，但还不能直接命名成门、入口或触发边界。

### 10. `.m byte08-15` 已经形成可视化与初步判断

- 已新增 `tools/analyze_m_byte_fields.py`，按 `.m` 单个 cell 的原始 `byte08-15` 绘制原始网格图、所有非零 value group 分色覆盖图，并输出 `derived/m_byte_fields/<stage>/m_byte_fields.json`。
- `byte08` 主要出现在陆地/水面交界与码头一类切换点，用于在陆地模型与水上模型之间切换；但仍有少量码头没有被它显式标记，因此它更像高价值辅助标记，而不是唯一真值源。
- `byte09` 已可确认为通行标记：`0` 可通行、`1` 不可通行；城墙、山寨、水陆交界、森林、山石等不可通行区域都会用到它。
- `byte10` 已可确认为城池、山寨等建筑的触发范围，其中 `byte10=239` 表示城门。
- `byte11` 已可确认为进入城池、山寨、军寨的触发范围编号；不同城池/山寨会使用不同 id，同值仍表现为稳定聚类区域。
- `byte13` 已可确认为小地图渲染颜色索引；`byte12`、`byte14`、`byte15` 在当前全量样本中都保持为 0，可视作填充/保留字节。
- `stage01.dor` 与 `stage01.m` 的对比结果已经写入 `derived/dor_relationship/stage01/dor_m_overlap.json`：当前 `.dor` 候选点位与 `.m byte08` 非零区的最佳精确重合只有 `1/38`，而与 `byte10!=0` / `byte11!=0` 的最佳精确重合可达 `26/29` / `24/29`；当前没有证据支持 `.dor` 直接记录码头/水陆交界切换点。

## 当前最值得继续推进的方向

1. 继续拆 `.stg` 内剩余未命名字段。
2. 把 `.evt` 的记录类型、坐标引用和对象引用正式化。
3. 从 `Emperor.exe` 继续收口 `.dor` 与 `.s/.x` 的真实读取/生成链路，解释为什么 `.x` 比 `.s` 更接近 `.m` 派生结果。
4. 把地图编辑器与 `.stg/.s/.x` 的联动补齐。
